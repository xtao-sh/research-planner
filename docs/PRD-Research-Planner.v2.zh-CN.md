# Research Planner v2 — 产品文档（PRD）

> **修订说明**
> 本文档是基于实际使用反馈的重大重新定位。v1 把产品定义为"面向学术研究的、关键路径式的项目排程工具"——这个定位在落地中暴露了根本错位：用户的真实需求不是把每个子任务排到日历上，而是**追踪进度、避免遗忘、积累项目相关的零散思考**。
>
> v2 把产品重新定位为：
>
> > **一个为不确定性而设计的个人多项目进度跟踪 + 项目知识库。**
>
> v1 文档保留为历史参考（`PRD-Research-Planner.zh-CN.md`）。v2 之后以本文为准。

---

## 1. 产品愿景

### 一句话定位
**当任务的耗时本身就不确定，进度比截止日期更重要。**

### 三个核心信念
1. **进度优先于截止**：截止日期是例外，不是默认。任务有"我现在在做什么 / 卡在哪 / 下一个做什么"的状态，比"必须 X 月 Y 日完成"更贴近研究和探索性工作的真实节奏。
2. **捕捉先于结构化**：好的工具应该**让用户先把想法快速丢进来**，再选择性地整理。强制结构化（必填截止、必填估时、必填依赖）是认知摩擦，会导致用户根本不打开工具。
3. **项目重启时给上下文**：研究项目经常被搁置数天到数月。重新打开一个项目时，工具应该立刻告诉用户"你上次做到哪、想到哪、什么是悬而未决的"，而不是让用户从一堆未排序的任务里自己回忆。

## 2. 目标用户与场景

### 用户画像
**一个同时维护多种性质项目的个人**——典型如：研究生、博士后、独立研究者、跨职能从业者、自由职业者。

用户的工作横跨至少三类：
- **研究类**（thesis、paper、综述、复现、实验）—— 周期长、产出不可预测、思考密集。
- **日常类**（学习一个新工具、读一本书、健康习惯、整理）—— 不紧急但会被遗忘。
- **行政类**（报销、申请、会议筹备、文档维护）—— 截止日期清晰但杂碎。

这三类共享一个属性：**单个任务的耗时高度不确定**，常规的"截止 + 估时"工具反而帮倒忙。

### 典型场景
1. **早上打开工具**：想知道"我今天该做什么"。系统呈现：当前进行中（doing）的任务、卡住（blocked）的任务、自己昨天选定的"今日聚焦"。**不**呈现一长串无差别的待办。
2. **写代码或看论文时突然想到一个点**：希望用 1 秒钟把它记录到对应项目下，**不需要**离开当前工作流去填表单。
3. **打开一个搁置两周的项目**：立刻看到"上次完成了什么、写了什么笔记、什么任务卡着"。
4. **周末回顾**：看一眼这周完成了什么、哪些项目在推进、哪些项目长期没动。
5. **某个项目有真实截止日期**（投稿 deadline、IRB 续审）：单独切到 deadline 模式，启用甘特/排程辅助。

### 不在范围内
- 多人协同的复杂权限矩阵（已实现，但不是默认 UX）。
- 资源/带宽分配、容量规划。
- 商业项目管理（PMP / Microsoft Project 风格的 critical-path 排程）—— 这些功能在代码里存在，UI 里需要降级到"高级"。

## 3. 设计原则

| 原则 | 含义 | 反例 |
|---|---|---|
| **进度 > 截止** | 默认追踪状态而非到期日 | 强制每个任务必填 due date |
| **灵活 > 精确** | 估时是可选；状态变化时间自动记录 | 强制 O/M/P 三个数字才能创建任务 |
| **捕捉 > 结构** | 1 秒能记下来一个想法 | 想法只能附加到已有任务上 |
| **简单默认 + 可选高级** | 第一次进入只看到必要字段；进阶功能藏在 toggle 后 | 把所有 PM 字段一次性铺开 |
| **重启给上下文** | 项目页打开时显式告诉用户"上次状态" | 让用户自己翻历史 |
| **单人优先** | 默认 UX 假设是一个人；协作是叠加层 | 默认显示成员管理、邀请、角色 |

## 4. 双模式设计

应用提供两种使用模式。**项目级别**切换。

### 4.1 进度模式（Progress mode）—— 默认
- 主视图是 **看板（Kanban）**：列 = 状态（todo / doing / blocked / done）。
- 不显示甘特、依赖图、关键路径、工作日历。
- 任务字段简化：标题、状态、尺寸（XS/S/M/L/XL）、可选标签、可选硬截止、备注。
- 估时（O/M/P）隐藏；尺寸是默认。
- 适用：90% 的研究任务、日常任务。

### 4.2 截止模式（Deadline mode）—— 按需开启
- 主视图增加甘特和依赖。
- 工作日历、节假日、关键路径、PERT 期望值全部生效。
- 任务字段扩展：估时（O/M/P）、依赖类型 + lag。
- 适用：投稿冲刺、申请截止、有外部硬约束的阶段。

**切换**：项目设置里一个开关；切换不改变数据，只切换 UI 复杂度。

## 5. 信息架构

### 5.1 一级路由（主导航）

| 路由 | 名称 | 用途 |
|---|---|---|
| `/now` | 今日聚焦 | 默认着陆页。当前进行 / 阻塞 / 今日选定 / 一行行动日志 / 全局快捷捕捉 |
| `/projects` | 项目画廊 | 项目卡片列表。每张卡显示进度概览 + 滞留指示 + 类型标签 |
| `/projects/:id` | 项目详情 | 任务、笔记、工件、时间线 4 个 tab。**整个 app 的核心**。 |
| `/inbox` | 收件箱 | 暂未归属项目的捕获项的整理界面 |
| `/review` | 回顾 | 周/月回顾（重定位为"反思"，不是"完成率报表"） |
| `/search` | 搜索 | 跨项目全文 + 标签 |

### 5.2 项目详情页 4 个 tab

1. **任务（Tasks）**：默认 Kanban；列表为辅。Gantt/依赖只在截止模式下出现。
2. **笔记（Notes）**：项目知识库的核心。每条 = 时间戳 + 内容 + 可选标签 + 可选关联任务。Markdown 轻量渲染。
3. **工件（Artifacts）**：链接、文件引用、代码片段、文献条目。
4. **时间线（Timeline）**：项目的活动历史（即现有的 event log，重命名为面向用户的"日志"）。

### 5.3 高级功能（藏在每个项目的"高级"区域，或全局设置中）

- 工作日历 / 节假日（仅 deadline 模式有效）
- 场景对比 / 备选方案（v1 的 scenarios）
- 计划分支（v1 的 branches，疑似可砍）
- 工作区成员管理（仅当工作区有 >1 成员时露出）
- 角色权限管理
- 邀请生成

### 5.4 顶部操作栏（在每个页面都可用）

- 全局快捷捕捉按钮（也可热键 `Cmd/Ctrl+Shift+N`）
- 项目切换器
- 语言切换
- 在线指示
- 退出登录

## 6. 数据模型（核心 vs 高级）

> 模型本身基本不变，但**字段在 UI 上分层**。代码里所有字段保留。

### 6.1 核心实体

```
Project { id, name, description, type, mode, createdAt, lastTouchedAt, color? }
Task    { id, projectId, title, status, size, tags?, notes?, dueHard?, startedAt?, finishedAt? }
Note    { id, projectId, taskId?, body (markdown), tags?, createdAt }
Artifact { id, projectId, kind, url, title?, tags?, createdAt }
```

### 6.2 高级实体（仅 deadline 模式 / 高级 UI 露出）

```
Estimate    { o, m, p, confidence }    -- 在任务的"高级"折叠区
Dependency  { fromTaskId, toTaskId, type, lag }   -- 仅 deadline 模式
Milestone   { ... }                    -- 仅 deadline 模式
Scenario    { ... }                    -- 高级
Calendar    { weeklyHours, holidays }  -- 仅 deadline 模式
```

### 6.3 协作层（已实现，UI 默认隐藏）

```
Workspace, WorkspaceMember (roles), Invite, Event, Session, User
```
单人使用时不出现。当用户主动邀请第二个成员或切换工作区时才浮现。

### 6.4 新增字段建议

- `Project.type`：枚举 `research / daily / admin / personal / other`，用户可扩展。决定模板和颜色。
- `Project.mode`：枚举 `progress | deadline`，决定项目页的 UI 复杂度。
- `Task.size`：枚举 `xs | s | m | l | xl`。比 O/M/P 轻量百倍，主用。
- `Task.startedAt / finishedAt`：状态变化时自动记录，用于"耗时"统计而非估时。
- `Project.lastTouchedAt`：用于项目画廊的"滞留"指示。

## 7. 关键功能

### 7.1 进度跟踪（Goal 1）

#### 7.1.1 状态是一切
四个状态：`todo / doing / blocked / done`。`review` 在单人场景里基本无意义，可选保留。

#### 7.1.2 尺寸代替精确估时
默认字段是 `size`：XS / S / M / L / XL（或 1/2/3/5/8 Fibonacci）。**O/M/P 移到任务卡的"高级"折叠区**。

#### 7.1.3 WIP 限制
每个项目可以选填一个"同时进行中上限"（默认 3）。超过时不会硬阻塞，但会显示警告："你已有 5 个 doing，先完成几个再开新的？"

#### 7.1.4 状态时间自动记录
- `todo → doing` 时记录 `startedAt`。
- `doing → done` 时记录 `finishedAt`。
- `doing` 状态总时长 = sum of (finishedAt - startedAt)。
- 一段时间后产生**实际尺寸→实际时长**的统计，比让用户瞎猜 O/M/P 准得多。

#### 7.1.5 滞留警告
- 任务在 `doing` >7 天：黄色 "stale"。
- 任务在 `blocked` >3 天：红色 "needs unblock"。
- 项目无任何 `lastTouchedAt` 更新 >14 天：项目卡显示"休眠中"。

#### 7.1.6 今日聚焦
`/now` 页面允许用户每天**主动选定** 1-5 个任务作为今日重点。与 doing 状态正交：可以选择一个 todo 任务作为今日聚焦。

#### 7.1.7 每日 / 每周回顾提示
- 每日：可选的"打卡"——昨天完成什么、今天打算做什么、有什么阻塞。3 行内。
- 每周：自动生成的回顾草稿（重定位现有的 ReviewReport）：完成了什么、哪些项目在推进、哪些休眠、哪些卡了一周。

#### 7.1.8 硬截止仍然支持
- 任务可填 `dueHard`（可选）。
- 即使在 progress 模式，硬截止违反会标黄/标红。
- 如果项目里**任何**任务有 `dueHard`，自动建议切到 deadline 模式（不强制）。

### 7.2 项目知识库（Goal 2）

#### 7.2.1 全局快捷捕捉
- 任意页面都可见的"+ 想法"按钮。
- 全局热键 `Cmd/Ctrl+Shift+N`。
- 点击或按下后弹出极简文本框：内容 + 项目（默认当前正在看的项目，可选其他、可选 inbox）+ 可选 #标签（自动从文本提取）。
- 一键保存。回车 = 保存并关闭。

#### 7.2.2 笔记是项目内的一等公民
- 项目详情页 4 个 tab 里，"笔记"和"任务"并列。
- 笔记可以是独立想法，也可以挂到某个任务上作为上下文。
- 笔记支持 markdown 轻渲染。
- "把这条笔记升格为任务"一键操作。

#### 7.2.3 项目重启简报
当用户打开一个 `lastTouchedAt > 3 天前`的项目时，详情页顶部出现一张"上下文卡"：
- "你上次活动：完成任务「X」/ 写了笔记「Y」"
- "悬而未决：阻塞中的「Z」、近 7 天没动的 doing 任务"
- "最近一条想法：『…』"

可关闭，下次打开同项目（无新活动）不再出现。

#### 7.2.4 标签自动抽取
笔记/任务正文里写 `#某主题` 自动入库为标签。点击标签 = 跨项目搜索同标签内容。

#### 7.2.5 反向链接（V3）
笔记里写 `[[其他笔记标题]]` 自动建立双向链接。建立项目内的小型知识图谱。**短期不做**——但数据模型应预留。

#### 7.2.6 收件箱（Inbox）
快捷捕捉时如果不选项目，落到全局 inbox。`/inbox` 页面用于事后整理：每条想法分配项目 / 标签 / 升格为任务 / 归档。

## 8. UI 重排（v1 → v2 落地）

### 8.1 上调到主 UX
- 状态 / Kanban
- 笔记 tab（升级为一等公民）
- 全局快捷捕捉
- 今日聚焦 (`/now` 页)
- 项目重启简报
- 滞留指示
- 状态时间统计

### 8.2 降级到"高级"或仅 deadline 模式可见
- 甘特图 / 关键路径
- 依赖图 / 依赖类型 + lag
- 工作日历 / 节假日编辑
- 估时（O/M/P）
- 场景保存与对比
- 计划分支
- 决策点（与分支耦合，可一并降级）

### 8.3 重命名 / 重定位
- "学术研究日程表" → "Research Planner — 进度日志"
- "研究日志（Research Journal）" → "笔记（Notes）"
- "活动（Activity）" → "时间线（Timeline）"
- "回顾报告（Review Report）" → "进度回顾（Progress retrospective）"——以反思为主，去除"完成率报表"的味道
- "工件（Artifact）" 保留，但放到 tab 里而非弹窗

### 8.4 默认隐藏（仅在多用户工作区或主动开启时浮现）
- 成员面板 / 角色 / 邀请
- 在线 presence
- 工作区切换器（默认显示"个人"工作区，无切换 UI）

### 8.5 砍除候选
- **计划分支（branches）**：和现实工作流脱节，建议从 UI 移除（代码暂时保留，等 v3 评估）。
- **决策点（decisions）**：和分支强耦合，一并下架。

## 9. MVP（v2）重新定义

v2 的"MVP"不是"做出一个完整 PM 工具"，而是"在 4-6 周内让用户的真实日常需求能跑起来"。

### 9.1 必须有
- `/now` 页面（今日聚焦 + 阻塞清单 + 全局快捷捕捉入口）
- 项目画廊（带滞留指示）
- 项目详情：任务（Kanban）+ 笔记 + 工件 + 时间线
- 全局快捷捕捉（按钮 + 热键）
- 收件箱（`/inbox`）
- 双模式开关（progress / deadline）
- 状态时间自动记录
- 项目重启简报
- `Project.type` 字段（决定颜色和模板）
- `Task.size` 字段（替代 O/M/P 默认）

### 9.2 已有但要降级 / 调整
- Kanban 升为主视图，Gantt/Dep 收到"高级"折叠
- O/M/P 收到"高级"折叠
- 工作日历仅在 deadline 模式可见
- 成员面板默认隐藏
- 主导航重排

### 9.3 不在 v2 MVP
- 反向链接 / 知识图谱
- 文献集成（Zotero / OSF / GitHub）
- 蒙特卡洛仿真
- 移动端
- 离线 / PWA
- 双向日历同步

## 10. 路线图（v2 → v3）

### v2.0（当前 PR D）—— 4-6 周
完成第 9 节列出的所有内容。落地后用户应该能：
- 早上打开 `/now`，5 秒内知道今天该做什么。
- 看完一篇论文后 1 秒钟丢一个想法到对应项目。
- 打开搁置 2 周的项目，立刻看到上次的状态和悬而未决项。
- 切到 deadline 模式只在投稿前最后两周用。

### v2.5 —— 增量
- 反向链接 (`[[note]]`)
- 项目模板系统（research / daily / admin 各一套预设）
- "今日聚焦"的智能建议（从 doing + 滞留 + 用户历史中挑）
- 简易日历视图（不是甘特，是月历，仅显示有 `dueHard` 的任务）

### v3 —— 长期
- 文献集成（Zotero）
- 移动端 PWA
- 离线优先
- 跨设备同步（已经有 WS，扩展为冲突合并）
- LLM 辅助：项目重启简报自动生成 / 笔记自动归类

### v2.0 实施分期（基于 §13 决策）

| 阶段 | 主题 | 预估 | 关键交付 |
|---|---|---|---|
| **A — 砍枝** | 砍除 branches + decisions；项目类型字段 | 2-3 天 | `BranchManager` UI 删；branch state / events / journal `decision` type 删；新增 `Project.type` 字段 + 5 种 type-preset 色板；现有项目默认 `other` |
| **B — 主导航** | 引入 `/now` `/inbox`；改默认着陆页 | 4-6 天 | 真正的路由（目前 App.tsx 单页内 viewMode 切换）；`/now` 显示当前 doing / blocked / 今日聚焦；`/inbox` 列出未归项目的捕捉条；项目画廊改造（带滞留指示） |
| **C — 捕捉** | 全局快捷捕捉 + Inbox 三联流 | 3-4 天 | 浮动"+"按钮 + `Cmd/Ctrl+Shift+N` 热键；最小弹层（textarea + 项目选择器）；写入项目 notes 或 inbox；inbox 整理 UI |
| **D — Size 化** | 任务尺寸代替 O/M/P 默认 | 2-3 天 | `Task.size` 字段；任务卡显示尺寸标签；O/M/P 隐藏到"高级"折叠；现有任务从 `m` 估时 backfill 一个粗略 size |
| **E — 笔记一等公民** | 项目详情 4-tab 改造 | 4-5 天 | 项目页：任务 / 笔记 / 工件 / 时间线 4 个 tab；现有研究日志改名为 notes 并迁出弹窗；任务-笔记关联；# 标签自动抽取 |
| **F — 重启简报 + 滞留 + 时长** | 进度跟踪闭环 | 3-4 天 | 项目页顶部"上下文卡"；doing >7d / blocked >3d / project 14d 滞留色阶；状态变化自动写 startedAt / finishedAt |
| **G — 双模式开关** | Progress / Deadline 切换 | 2 天 | 项目设置一个 toggle；deadline 模式下露出 Gantt + 工作日历 + 依赖类型；progress 模式下隐藏 |
| **H — 隐藏单人不需要的协作 UX** | 默认 UX 简化 | 1-2 天 | 工作区有 1 人时，成员面板 / 邀请 / 在线指示从 UI 上隐藏；切换工作区或邀请第一个成员后重新出现 |
| **I — 重命名 + 回顾重定位** | 文案与定位调整 | 1-2 天 | "活动 → 时间线"；"研究日志 → 笔记"；review 报告改为反思视角；导出报告对应改 |

**总估计**：22-31 天工作日（约 5-6 周）。

阶段相对独立，可以串行也可并行（A 不依赖任何东西，是首选起点；B 是后续所有 UX 的基础）。每个阶段完成后跑全量测试 + 一次 commit。

## 11. 非功能需求

（与 v1 一致，扼要）

- 性能：1-5k 任务流畅；笔记 markdown 渲染懒加载。
- 可靠性：本地缓存 + 服务端持久化；版本恢复（基于现有 event log）。
- 可用性：键盘优先；快捷捕捉 1 次操作可达；暗色模式；i18n（中/英）。
- 隐私：默认私有；分享通过链接显式开启；敏感字段脱敏。

## 12. 成功指标（重新定义）

v1 用了"周活跃 / 完成率 / 缓冲消耗"等 PM 指标。v2 应该度量**用户是否真的少遗忘、多沉淀**：

| 指标 | 目标 |
|---|---|
| 每日打开 `/now` 的天数 | 越高越好（习惯养成） |
| 通过快捷捕捉创建的笔记数 | 越高越好（捕捉摩擦低） |
| 项目重启简报点击查看率 | >60%（说明有用） |
| 任务在 `doing` 滞留中位数 | 监测——上升说明"做不完"；下降说明状态被维护 |
| 笔记 → 任务 升格次数 | 表示用户真的回头消费自己的笔记 |
| Inbox 处理时长 | 中位数 <2 天（处理及时） |

去除 v1 里"准时完成率""排程重排响应时间"等指标——它们在进度模式下没意义。

## 14. 与 v1 PRD 的差异概览

| 主题 | v1 | v2 |
